# Exec Plans 작성 규칙

> 모든 비자명한 기능 개발·리팩토링·테스트 작업은 여기에 계획을 먼저 기록합니다.
> 계획에 없는 작업은 즉흥 작업이며, 의사결정이 휘발됩니다.

---

## 디렉터리 구조

| 경로 | 용도 |
|------|------|
| `active/` | 진행 중인 계획. 머지 전까지 여기에 둠 |
| `completed/` | 완료된 계획. 의사결정 로그 보존용 — 삭제 금지 |
| `tech-debt-tracker.md` | 알려진 기술 부채 추적 (별도 파일) |

---

## 기본 양식

새 계획을 만들 때 아래 4개 섹션을 반드시 포함합니다.

```markdown
# [상태] 계획 제목

- **시작일** 또는 **완료일**: YYYY-MM-DD
- **브랜치**: feature/xxx_누구
- **해결한 부채** (해당 시): TD-XXX

## 목표
무엇을, 왜 하는가. 한 문단.

## 수용 기준
- [ ] 체크박스로 측정 가능한 완료 조건
- [ ] 빌드·테스트가 통과한다 등 검증 가능한 항목

## 의사결정 로그
- 결정 사항과 이유를 시점 순으로 기록
- 대안 검토 결과도 함께 기록 (왜 X를 선택하지 않았는가)
```

---

## 테스트 계획 추가 규칙

테스트 작성 계획에는 **테스트 케이스 표**를 반드시 포함합니다.
케이스 추가·삭제가 발생하면 계획 문서도 같이 갱신합니다.

### 필수 컬럼

| 컬럼 | 설명 |
|------|------|
| `#` | 번호 |
| `메서드` | `@Test` 메서드명 (코드와 1:1 매칭) |
| `검증 내용` | 무엇을 입력/조건으로 주는가 |
| `기대 결과` | 무엇이 일어나야 하는가 (반환값·예외·side effect) |

### 예시

```markdown
### UserAuthServiceTest (11)

| # | 메서드 | 검증 내용 | 기대 결과 |
|---|--------|----------|----------|
| 1 | `login_success` | 유효한 자격증명 | 토큰 페어 발급 + RefreshToken 저장 |
| 2 | `login_userNotFound` | 존재하지 않는 이메일 | `USER_NOT_FOUND` |
| 3 | `login_softDeletedUser` | 탈퇴(soft-delete) 유저 | `USER_NOT_FOUND` |
```

전체 예시: [completed/service-unit-tests.md](completed/service-unit-tests.md)

---

## 작성 시점

| 시점 | 행동 |
|------|------|
| 작업 시작 전 | `active/`에 계획 파일 생성. 수용 기준까지 채우고 시작 |
| 작업 도중 | 의사결정 로그 추가. 수용 기준이 바뀌면 함께 갱신 |
| PR 머지 시 | `active/` → `completed/`로 이동, 완료일·PR 기록 |
| 해당 작업이 부채를 해소 | `tech-debt-tracker.md`의 ✅ 해결 완료 표에 행 추가 |
